# Compiler build profiling

JFR-based profiling of the compiler build itself, to find compile-time
hotspots (e.g. in the monomorphization/specialization passes) and to catch
performance regressions as features land.

The process being profiled is the **boot compiler** — it is the JVM that runs
`--build idris2.ipkg`. To measure the cost of changes on your branch, the
flow is:

1. Build the compiler with your changes and install it as the boot compiler
   (or point `IDRIS2_BOOT` at it).
2. `./profiling/profile-build.sh -l my-change`
3. `./profiling/compare.py results/<baseline> results/<new-run>`

## Quick start

```sh
# Warm build (default): re-runs only the whole-program JVM codegen, where the
# monomorphization passes run. Fast; the right mode for regression checks.
./profiling/profile-build.sh -l baseline

# Full clean build (slow, also measures typechecking all modules):
./profiling/profile-build.sh -c -l clean-baseline

# Timing only, no JFR. Deep-stack sampling inflates wall clock (~80%
# observed), so compare wall times between -t runs and use profiled runs
# only to attribute the time:
./profiling/profile-build.sh -t -l timing-baseline

# Compare two runs (exits 1 on >10% wall-clock regression):
./profiling/compare.py profiling/results/<run-A> profiling/results/<run-B>
```

Requires JDK 21+ for the `jfr` tool (JDK 25 in use), and `python3`.

## What you get

Each run writes `profiling/results/<timestamp>-<sha>[-label]/`:

- `report.md` — wall time, phase split, hot-method tables, GC summary
- `report.json` — same data, consumed by `compare.py`
- `build.jfr` — raw recording; open in JDK Mission Control for flame graphs,
  or explore with `jfr view <view-name> build.jfr`
- `build.log` — make output

`profiling/history.csv` accumulates one row per run (timestamp, sha, mode,
wall seconds, jvm-backend sample share) for long-term trend tracking.

## How the report attributes time

Execution samples are bucketed by walking each stack leaf-to-root until a
domain frame is found:

| Bucket | Frames | Meaning |
|---|---|---|
| `jvm-backend` | `M_Compiler.M_Jvm.*`, assembler, ASM | monomorphization, type inference, codegen, bytecode assembly |
| `backend-common` | other `M_Compiler.*` | shared backend (CompileExpr, inlining, ANF) |
| `frontend` | `M_Core.*`, `M_TTImp.*`, `M_Idris.*`, `M_Parser.*` | parsing, elaboration, typechecking |
| `runtime/other` | nothing matched | runtime helpers, GC, I/O, JIT |

"Inclusive" tables count a sample for every distinct method anywhere in its
stack, so a method's % reads as "fraction of build time spent inside this
function or its callees". The `Top Compiler.Jvm.*` table is the one to watch
for monomorphization regressions (`inferDef`, `rewriteSpecCalls`,
`buildSpecialisationPlan`, ...).

## Notes

- Warm-mode numbers are comparable to other warm runs only; clean to clean.
  (`src/IdrisPaths.idr` is force-regenerated by make, so even a warm
  `make idris2-exec` re-runs the whole-program codegen every time.)
- The boot launcher hardcodes `-Xss92m -Xms3g -Xmx3g` *after* `JAVA_OPTS`,
  so heap size cannot be changed from here; check the GC section of the
  report to see whether the build is GC-bound.
- `-c` deletes `build/ttc` and `build/exec`.
